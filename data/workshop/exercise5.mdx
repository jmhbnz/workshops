## 5.1 Execute an arbitrary process

Before we introduce any runtime controls, let's first prove an attacker (or anyone with cluster access) can execute arbitrary processes inside a running container.

We'll use the `payment-gateway` pod from our incident scenario. Run the following command to exec into the container and launch a shell:

Remember to set your `namespace:` in the command below before executing it within your web terminal.

```bash
oc --namespace payment-gateway-x exec -it pod/payment-gateway -- /bin/bash -c "whoami && hostname && echo 'Arbitrary process execution successful'"
```

You should see output confirming the shell ran successfully, displaying the user and hostname inside the container. This is exactly the kind of capability a threat actor exploits after gaining initial access. Once they can execute arbitrary processes, they can download toolkits, exfiltrate data, or establish persistence.

> **Morgan**: See that? Anyone with exec permissions can drop into a shell and run whatever they want. The network policy improvements we've been rolling out buys us time, but it doesn't stop code execution inside the container. We need something that catches this at the process level, and does so automatically.

> **Sarah**: Agreed. Let's look at what RHACS can do for us here.


## 5.2 Introduce process baselining

Red Hat Advanced Cluster Security (RHACS) uses [eBPF](https://ebpf.io/) (extended Berkeley Packet Filter) probes on each cluster node to monitor every process that starts inside a container in real time, with zero instrumentation required in the application itself. It's actually an embedded instance of the [Falco](https://github.com/falcosecurity/libs) eBPF probes.

When a deployment is first observed, RHACS enters an **observation window** where it records every process that runs inside each container. This set of observed processes forms the **process baseline**. Think of it as a representation of "normal" behaviour for that container.

While the baseline is **unlocked**, RHACS continues to add newly observed processes to the baseline it's learning. Once the baseline is **locked**, any process execution that falls outside the baseline is flagged as an **anomalous process** violation.

This gives platform and security teams an extremely powerful runtime control, based on behavioural analysis: even if a vulnerability allows remote code execution, RHACS will detect and alert on the anomalous process that the exploit attempts to run.

Let's see this in action. Let's return to the [RHACS web console](https://central-stackrox.apps.cluster-sr857.sr857.sandbox1037.opentlc.com/) and navigate to **Risk** from the left-hand menu. Search for the `payment-gateway` pod.

Click on the pod to open its risk details, then select the **Process Discovery** tab. You'll see the list of processes that RHACS has observed running in the container.

Notice that the baseline is currently **unlocked**. RHACS is still in its observation window and will continue adding any new processes it sees to the baseline. This means our earlier `exec` command may have been captured, but it won't trigger a violation because the baseline hasn't been enforced yet.

> **Morgan**: So right now it's just watching and learning?

> **Sarah**: Exactly. Once we lock the baseline, anything not on the list gets flagged. Let's do that now.


## 5.3 Lock a baseline

Now we'll lock the process baseline for the `payment-gateway` container. Once locked, only the processes already in the baseline will be considered "normal". Everything else triggers a violation.

In the RHACS web console:

1. Navigate to **Risk** and select the `payment-gateway` deployment
2. Click the **Process Discovery** tab
3. Review the list of baselined processes, these should be the legitimate application processes (e.g., the `httpd` web server and its worker processes)
4. If you see `/bin/bash` or other shell processes in the baseline from our earlier exec, **remove them** by clicking the **X** next to the process entry. We don't want shells to be considered normal behaviour
5. Click the **Lock** icon (🔒) to lock the baseline

With the baseline locked, let's prove it works. Run the same exec command we used earlier:

```bash
oc --namespace payment-gateway-x exec -it pod/payment-gateway -- /bin/bash -c "whoami && hostname && echo 'Can I still do this?'"
```

The command itself will still execute (RHACS is **detecting**, not **blocking** by default), but now navigate back to the RHACS web console and check the **Violations** page from the left-hand menu.

You should see a new violation: **Unauthorized Process Execution** for the `payment-gateway` deployment. Click on it to see the details. RHACS has recorded that `/bin/bash` was executed and it was not in the locked baseline.

## 5.4 Automate locking of baselines for entire namespace

Locking baselines one deployment at a time through the web console is fine for a targeted change, but ACME has hundreds of deployments. Morgan wants every workload in the `payment-gateway` namespace locked down before end of day.

Fortunately, RHACS exposes a full [REST API](https://central-stackrox.apps.cluster-sr857.sr857.sandbox1037.opentlc.com/main/apidocs) that we can use to automate this. The API includes a [bulk lock endpoint](https://github.com/stackrox/stackrox/blob/master/proto/api/v1/process_baseline_service.proto) that can lock all process baselines for a given cluster and namespace in a single call.

First, generate an API token in the RHACS web console:

1. Navigate to **Platform Configuration** → **Integrations**
2. Scroll down to **Authentication Tokens** and click **API Token**
3. Click **Generate Token**, give it a name like `baseline-automation`, and select the `Admin` role
4. Copy the generated token

Now set your environment variables:

```bash
export ROX_API_TOKEN="<your-token-here>"
export ROX_CENTRAL_ADDRESS="central-stackrox.apps.cluster-sr857.sr857.sandbox1037.opentlc.com:443"
```

We need the cluster ID for the bulk lock request. Retrieve it with:

```bash
curl --silent --insecure \
  --header "Authorization: Bearer ${ROX_API_TOKEN}" \
  "https://${ROX_CENTRAL_ADDRESS}/v1/clusters" \
  | jq -r '.clusters[] | "\(.id) \(.name)"'
```

Set the cluster ID and namespace from the output above:

```bash
export CLUSTER_ID="<your-cluster-id>"
export NAMESPACE="vulnerable-workload-x"
```

Now lock all process baselines in the `vulnerable-workload-x` namespace with a single API call:

```bash
curl --silent --insecure \
  --header "Authorization: Bearer ${ROX_API_TOKEN}" \
  --header "Content-Type: application/json" \
  --request PUT \
  "https://${ROX_CENTRAL_ADDRESS}/v1/processbaselines/bulk/lock" \
  --data "{\"cluster_id\": \"${CLUSTER_ID}\", \"namespaces\": [\"${NAMESPACE}\"]}" | jq .
```

You should see `{"success": true}` confirming all baselines in the namespace have been locked. Navigate back to the RHACS **Risk** view and verify that the baselines for all deployments in the `vulnerable-workload-x` namespace now show as locked.

> **Sarah**: And the best part is we didn't have to change a single line of application code. The platform is doing the heavy lifting.
