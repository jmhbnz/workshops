With the ACME health check pipelines established, Sarah wants to handle a common day-2 operational issue: applications running out of disk space because she's tired of being paged out for this when on call! 😩

![Sarah paged out](/workshops/static/images/ocp-aap/sarah-callout.png)

Instead of being called out and needing to expand storage, she wants configure the platform to automatically expand persistent volumes using Event-Driven Ansible (EDA) when a Prometheus monitoring alert fires due to storage reaching a particular threshold.

Of course a manual incident still needs to be raised in ServiceNow so the team responsible for the service can look into the root cause and take action, but Sarah is determined to put an end to being called out for this issue when more storage can be easily provisioned automatically and teams can look at the issue later during business hours.

> **Background:** [Event-Driven Ansible](https://www.ansible.com/use-cases/event-driven-automation) (EDA) can ingest real-time events from external sources like OpenShift's Prometheus Alertmanager stack. When an alert fires indicating a volume is reaching capacity, EDA intercepts the alert payload, processes the metadata, and immediately fires a playbook to resize the target volume before an outage occurs.


## 5.1 - Creating the application workload

To build a self-healing loop for a storage emergency, Sarah first needs to provision a mock application environment that utilizes a persistent volume claim (PVC) inside the cluster.

Let's stand up this environment now.

Click the **+ Quick Create** button in the top right corner of the OpenShift web console, then click **Import YAML**

Copy the content below to deploy the sample workload and then click **Create**.

```yaml
---
apiVersion: v1
kind: Namespace
metadata:
  name: production

---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: acme-db-pvc
  namespace: production
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: acme-database
  namespace: production
spec:
  replicas: 1
  selector:
    matchLabels:
      app: acme-database
  template:
    metadata:
      labels:
        app: acme-database
    spec:
      containers:
        - name: database
          image: registry.access.redhat.com/ubi9/ubi-minimal:latest
          command: ["/bin/sh", "-c", "while true; do sleep 30; done"]
          volumeMounts:
            - name: data-volume
              mountPath: /var/lib/data
          resources:
            limits:
              cpu: "100m"
              memory: "128Mi"
      volumes:
        - name: data-volume
          persistentVolumeClaim:
            claimName: acme-db-pvc
```

 *Importing sample application*



Verify that the storage binds successfully and the database pod transitions into a healthy state by clicking **Storage** on the left hand menu, then click **PersistentVolumeClaims**.

Ensure the `production` namespace is selected from the **Project** dropdown.

Examing the `acme-db-pvc` volume we created we should see this is `Bound` to a running pod, with a starting capacity of `1Gi`.

## 5.2 - Creating an aap token

The event driven automation workflow Sarah is going to create is going to execute a job template within Ansible Automation Platform to remediate the storage issue.

To enable this, we're going to need to give the event driven automation a token to use to authenticate back to the platform and trigger the remediation job template.

Let's build this credential inside the web interface.

The runner container is now healthy and actively listening for data packets on internal port `5000`.

Take a moment explore the rulebook Sarah wrote earlier, notice how we are listing for alerts matching `PersistentVolumeUsageCritical`. When we identify one we will run the `acme-ocp-expand-volume` job template, which will automatically receive the event payload as a variable.

```yaml
---
- name: Listen for storage alerts from alertmanager
  hosts: all
  sources:
    - ansible.eda.webhook:
        host: 0.0.0.0
        port: 5000
  rules:
    - name: Match volume filling alert
      condition: event.payload.alerts[0].labels.alertname == "PersistentVolumeUsageCritical"
      action:
        run_job_template:
          name: acme-ocp-expand-volume
          organization: Default
```

## 5.7 - Configuring openshift alerting

With all the foundations in place within ansible, Sarah moves on to configuring OpenShift alerting to ensure alerts will be forward to the newly established event driven ansible listener.

Let's create an `AlertmanagerConfig` now.

Click the **+ Quick Create** button in the top right corner of the OpenShift web console, then click **Import YAML**

Copy the content below to deploy the sample workload and then click **Create**.

```yaml
apiVersion: monitoring.coreos.com/v1alpha1
kind: AlertmanagerConfig
metadata:
  name: eda-storage-routing
  namespace: production
spec:
  route:
    receiver: eda-webhook-receiver
    groupBy: ['alertname']
    matchers:
      - name: alertname
        value: PersistentVolumeUsageCritical
  receivers:
    - name: eda-webhook-receiver
      webhookConfigs:
        - url: http://acme-ocp-volume-expansion.aap.svc.cluster.local:5000/endpoint
```


![Create alertmanager config](/workshops/static/images/ocp-aap/ocp-alertmanager.png)
 *Creating alertmanager config*



## 5.8 - Triggering the event

With event driven ansible configured, and openshift alerting wired up Sarah is ready to give the workflow a try.

We can run a quick bash command within our demo workload pod to fill up the storage volume and trigger the `PersistentVolumeUsageCritical` alert our event driven ansible listener is waiting for.

Start by clicking **Workloads** in the OpenShift left menu, then click **Pods**.

Ensure the `production` project is selected from the project dropdown at the top.

Click in to your `acme-database` pod, then click the **Terminal** tab at the top.

Copy and run the following command in the terminal: `dd if=/dev/zero of=/var/lib/data/dump.img bs=1M count=850` which will create an 850Mb file on the 1Gi volume, enough to trigger the alert.


![Run pod command](/workshops/static/images/ocp-aap/pod-command.png)
 *Running command within pod*

Within a couple of minutes, the `PersistentVolumeUsageCritical` will begin firing within the cluster. Ansible will then be notifed. Our automatic remediation job template will be run, and the volume will be increased in size.

Navigate back to your volume to confirm it is now 2Gi.
