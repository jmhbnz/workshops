After yesterday's incident, ACME's Infosec lead Morgan has asked Sarah to prove that the team's container hardening strategy actually works. The best way? Find a real vulnerability, exploit it live, then show how hardened images neutralise the threat.

Sarah has been reading about [CVE-2025-61765](https://nvd.nist.gov/vuln/detail/CVE-2025-61765) — a remote code execution vulnerability in `python-socketio` versions prior to `5.14.0`. It's a textbook insecure deserialization flaw that lets an attacker run arbitrary code on any Socket.IO server that uses a shared message broker like Redis.

![image-hardening](/workshops/static/images/fsi-security/sarah-verification.png)


## 4.1 Understanding CVE-2025-61765

When multiple Socket.IO servers share a message broker for inter-server communication, messages are serialised using Python's `pickle` module. When a server receives a message from the broker it calls `pickle.loads()` automatically — trusting that the data is safe.

An attacker who can reach the message broker (or any service that publishes to it) can send a crafted pickle payload that executes arbitrary code during deserialization through Python's `__reduce__` method.

Here's the core of the exploit — a class whose `__reduce__` method tells `pickle.loads()` to call `subprocess.call()` with an attacker-controlled command:

```python
class RCEPayload:
    def __init__(self, cmd):
        self.cmd = cmd

    def __reduce__(self):
        return (subprocess.call, (["/bin/bash", "-c", self.cmd],))
```

When this object is unpickled by the vulnerable server, it runs the shell command. The attacker simply serialises this payload and publishes it to the broker channel that Socket.IO subscribes to.

> **Key insight:** The exploit depends on two things: (1) `pickle.loads()` deserializing untrusted data, and (2) a **shell** (`/bin/bash`) being available in the container image to execute the payload.


## 4.2 Deploy the vulnerable environment

Let's deploy a vulnerable Socket.IO application to our cluster. We'll start with a typical setup that many teams might have in production — a community Redis image as the message broker, and the application built on a standard UBI image with a full shell environment.

First, set up your personal namespace. Each participant gets their own namespace based on their username:

```bash
export NAMESPACE=ex4-$USER
```

The environment consists of:

- **Redis** — the message broker, using the community `docker.io/redis:7-alpine` image
- **Socket.IO server** — a Python application using `python-socketio==5.13.0` (the vulnerable version), built on a standard UBI image that includes a shell
- **Edge server** — a Flask webhook server that accepts HTTP payloads and publishes them to Redis

```bash
envsubst < setup/exercise4/k8s/namespace.yaml | oc apply -f -
envsubst < setup/exercise4/k8s/redis.yaml | oc apply -f -
envsubst < setup/exercise4/k8s/edge-server.yaml | oc apply -f -
envsubst < setup/exercise4/k8s/socketio-server.yaml | oc apply -f -
```

Wait for all pods to become ready:

```bash
oc -n $NAMESPACE get pods -w
```

You should see three running pods: `message-broker`, `socketio-server`, and `edge-server`.

## 4.3 Trigger the exploit

Now, let's act as the attacker. We'll run a Job that sends a malicious pickle payload to the edge server, which forwards it to the Redis broker, where the vulnerable Socket.IO server picks it up and deserializes it — executing our code.

```bash
envsubst < setup/exercise4/k8s/exploit-job.yaml | oc apply -f -
```

Watch the exploit Job's logs:

```bash
oc -n $NAMESPACE logs -f job/cve-2025-61765-exploit
```

You should see output confirming the payload was delivered to the edge server webhook and received by a subscriber.

Now check whether the exploit succeeded. The Socket.IO server exposes an `/evidence` endpoint that reads the evidence file. We can query it by running a verification Job that curls the endpoint:

```bash
envsubst < setup/exercise4/k8s/verify-job.yaml | oc apply -f -
oc -n $NAMESPACE wait --for=condition=complete job/verify-exploit --timeout=30s
oc -n $NAMESPACE logs job/verify-exploit
```

If the exploit worked, you'll see output like:

```
>>> COMPROMISED via CVE-2025-61765 <<<
```

**The Socket.IO server has been compromised.** An attacker delivered arbitrary code through a pickle payload — the server had no way to distinguish it from a legitimate inter-server message.

> Take a minute to review the exploit source at `setup/exercise4/exploit/poc.py` and the vulnerable server at `setup/exercise4/socketio-server/server.py`. The server code itself looks completely normal — the vulnerability is in the `python-socketio` library's use of `pickle.loads()`.


## 4.4 Introduce Project Hummingbird hardened images

[Project Hummingbird](https://www.redhat.com/en/technologies/linux-platforms/enterprise-linux/hummingbird) is Red Hat's catalog of minimal, hardened container images available at `registry.access.redhat.com/hi/`. These images are built with a *distroless* philosophy:

- **No shell** — no `bash`, `sh`, or any other shell binary
- **No package manager** — no `dnf`, `rpm`, or `apt`
- **Minimal attack surface** — only the runtime and its dependencies are included
- **Non-root by default** — containers run as a non-privileged user

This matters for CVE-2025-61765 because the exploit payload calls `subprocess.call(["/bin/bash", "-c", ...])`. If there's no `/bin/bash` in the container, the deserialization still happens but the **code execution fails** — there's nothing to execute.

Let's compare the two Containerfiles for the Socket.IO server:

**Vulnerable** (`Containerfile`) — built on UBI 9, which includes a full shell environment:

```dockerfile
FROM registry.redhat.io/ubi9/python-311
# ... has bash, coreutils, rpm, etc.
```

**Hardened** (`Containerfile.hardened`) — multi-stage build, runtime stage uses the Hummingbird distroless Python image. It also uses `requirements-hardened.txt`, which removes the version pin so `pip` pulls the latest (patched) `python-socketio`:

```dockerfile
FROM registry.access.redhat.com/hi/python:latest-builder AS builder
# install dependencies in builder (has shell)
# uses requirements-hardened.txt — no version pin on python-socketio

FROM registry.access.redhat.com/hi/python:latest
# runtime only — no shell, no package manager
COPY --from=builder /opt/venv /opt/venv
```
Before we harden the application, let's clean-up first:
```
oc -n $NAMESPACE exec deploy/socketio-server -- rm -f /tmp/evidence/pwned.txt 2>/dev/null; true
```

## 4.5 Harden the application — swap to the distroless Python image

Now let's address the exploit directly. Apply the hardened Socket.IO server deployment, which uses the Hummingbird distroless Python image:

```bash
envsubst < setup/exercise4/k8s/socketio-server-hardened.yaml | oc apply -f -
```

Wait for the new pod to roll out:

```bash
oc -n $NAMESPACE rollout status deploy/socketio-server
```

Verify the new pod is running the hardened image:

```bash
oc -n $NAMESPACE get deploy socketio-server -o jsonpath='{.spec.template.spec.containers[0].image}'
```

You should see `quay.io/smileyfritz/socketio-server:0.1.0-hardened`.


## 4.6 Verify the fix

First, delete the previous exploit and verification Jobs:

```bash
oc -n $NAMESPACE delete job cve-2025-61765-exploit verify-exploit --ignore-not-found
```

Now re-run the exploit:

```bash
envsubst < setup/exercise4/k8s/exploit-job.yaml | oc apply -f -
```

Watch the logs — the exploit will appear to send successfully (the edge server and Redis still relay the payload):

```bash
oc -n $NAMESPACE logs -f job/cve-2025-61765-exploit
```

Now run the verification Job to query the `/evidence` endpoint:

```bash
envsubst < setup/exercise4/k8s/verify-job.yaml | oc apply -f -
oc -n $NAMESPACE wait --for=condition=complete --for=condition=failed job/verify-exploit --timeout=30s
oc -n $NAMESPACE logs job/verify-exploit
```

The verification Job will **fail** — the evidence file was never created because the exploit payload called `subprocess.call(["/bin/bash", ...])` and there is no `/bin/bash` in the Hummingbird distroless image. The pickle deserialization still triggered, but the code execution failed silently.

> **Key takeaway:** The hardened application image didn't fix the vulnerable library code — `python-socketio 5.13.0` still calls `pickle.loads()`. But by removing the shell and minimising the image contents, the blast radius of the exploit is drastically reduced. Defence in depth means the vulnerability exists but **cannot be weaponised**.

### Clean up

```bash
oc delete namespace $NAMESPACE
```

Excellent work! You've seen how a real-world RCE can be exploited in a containerised environment, and how migrating from community images to Project Hummingbird's hardened images provides critical defence in depth even when application-level vulnerabilities exist.
