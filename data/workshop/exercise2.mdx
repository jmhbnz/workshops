## 2.1 - Identify the compromised credentials

Now online and frantically loading the [OpenShift Web Console](https://console-openshift-console.apps.cluster-6lrcm.dynamic2.redhatworkshops.io) for the `acme-prd` cluster Sarah keeps Morgan updated with her progress.

> **Sarah** Ok I'm logged in, do you want me to kill the pod? What do we do??

> **Morgan** Whatever you do, don't kill the pod yet!  We'll lose all the in memory evidence if we do that. We're monitoring the egress traffic and we'll isolate that shortly as our first actual counter move. For now I need you to collect some information. First up, how the hell are they executing this reverse shell? Is there anything you can see from your side on how this started Sarah??  We need to know if this is bigger than just one pod or not.

> **Sarah** Let me take a look in the OpenShift Audit Logs, give me a few minutes to hunt through the logs.

Throughout this workshop you'll be using a namespace assigned to your user number, for example `payment-gateway-1` for `user1`.

Let's review the audit logs, specifically for pods within your allocated `payment-gateway` namespace.  We can quickly query these via `oc` using our OpenShift Web Terminal. In an incident situation, fast response is critical, having the skills and tooling to be able to quickly query audit logs is immensely valuable.

Open a **Web Terminal** session using the **>_** button in the top right of the OpenShift Web Console.

<Zoom>
![web-terminal-button](/workshops/static/images/fsi-security/web-terminal-button.png)

Ensure your `payment-gateway-x` project is selected, then click **Start**.

<Zoom>
![web-terminal-project](/workshops/static/images/fsi-security/web-terminal-project.png)
*Select OpenShift Web Terminal project*

Run the bash script below which uses `oc adm` to query the audit logs held on our control nodes specifically for `exec` events occurring in the namespace of interest.

```bash
namespace=$(oc project --short)

for master in $(oc get nodes --selector node-role.kubernetes.io/master --output name); do
  echo "=== Searching logs on ${master} ==="
  oc adm node-logs "${master}" --path=kube-apiserver/audit.log \
    | jq --arg ns "${namespace}" 'first(select(.objectRef.namespace == $ns 
          and .objectRef.name == "payment-gateway"
          and .verb == "create" 
          and .objectRef.subresource == "exec"))'
done
```

> **Sarah:** (Rubbing her eyes, voice raspy) Morgan, I’m looking at the master node audit logs right now. Tell me you’re seeing this on my shared screen in Teams?

> **Morgan:** (Over the phone, typing audio in the background) I see it. I'm looking at the requestURI block right now... f&%$. Someone just spawned an interactive shell.

> **Sarah:** It’s worse than that. Look at the URL encoding in the query string. They didn't just open a terminal; they fed a full payload straight into the exec subresource. If you decode those hex characters, they're exporting a C2 server IP.

> **Morgan:** Wait, scroll back up... %2Ftmp%2F.hidden_toolkit%2Fexfil.sh. They're building a hidden persistence directory right inside the /tmp directory of our production gateway.

> **Sarah:** Exactly. And it worked. Look at the responseStatus.code. It’s a 101.

> **Morgan:** HTTP 101 Switching Protocols... the API server upgraded the connection to a live WebSocket. That means the attacker successfully established a persistent bidirectional stream directly into the httpd container. Who authorized this? Is it a compromised admin account?

## 2.4 - Isolate the workload

> **Sarah** Ok Morgan, I'm seeing what you are in terms of unexpected egress traffing from `payment-gateway`. Let's cut this traffic off right now with a network policy.

OpenShift supports software defined network segmentation through Kubernetes [`NetworkPolicy`](https://docs.redhat.com/en/documentation/openshift_container_platform/4.21/html/network_security/network-policy).  As a platform engineer, you can define network policies that restrict traffic to pods in your cluster. In an incident situation a NetworkPolicy is a critical tool to quickly adjust what network flows are possible.

Apply the default deny egress network policy by returning to the OpenShift Web Console and clicking the **Quick Create (+)** button in the top right corner. Ordinarly this type of change would be delivered through a GitOps methodology but given this is an active incident we are using break glass credentials and performing actions directly on the cluster.

Paste in the network policy example below, then update the `namespace:` to your user number.

Once updated, click **Create**.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-gateway-deny-egress
  namespace: payment-gateway-x
spec:
  # This targets only the payment-gateway
  podSelector:
    matchLabels:
      app: payment-gateway
  policyTypes:
    - Egress
  # Leaving egress empty here acts as a total block 
  # for any traffic originating from matching pods.
  egress: []
```

## 2.5 Another pair of eyes

During a security incident, `oc adm must-gather` can act as an easy to use forensic black box recorder, instantly capturing an unalterable snapshot of the cluster's state and history before critical data can be erased or rotated out. Gathering this comprehensive environment information right away is vital when engaging vendor support or external incident investigation suppport; it provides platform specialists with the exact telemetry they need to piece together how the breach occurred and verify the extent of a compromise.

With everything happening so quickly, after the short break Sarah opens a case with Red Hat support to confirm their initial findings around the use of a compromised credential and ensure she and Morgan did not miss anything.

Let's return to our web terminal and run the following command to retrieve the audit logs directly from the cluster, into a directory that can be easily zipped and attached to a case at [https://access.redhat.com/support/cases/#/case/new](https://access.redhat.com/support/cases/#/case/new). 

> **Important:** For this workshop we won't actually open a case, however if this were a real event, quickly engaging expert assistance is highly recommended and we should practice the process of retrieving logs in a sharable, structured format.

```bash
oc adm must-gather \
  --image=quay.io/openshift/origin-must-gather:latest \
  --run-namespace="$(oc project --short)" \
  --dest-dir=/tmp/incident-artifacts \
  -- /usr/bin/gather_audit_logs
```

Feel free to take a moment to explore the files that were just created on disk within your web terminal.
