The phone vibrated angrily on the wooden nightstand. It was 02:14 AM. Sarah, squinted at the caller ID: *ACME SOC Incident Team*.

> Sarah, It's Morgan from Infosec, sorry to call you out but we have a big f*%$@*^* problem, the voice on the other end said frantically. Our automated anomaly detection just flagged highly irregular egress traffic coming from the `payment-gateway` namespace on the `acme-prod` cluster! We think a threat actor has breached the container, you need to get online right now!!

Sarah threw open her laptop and immediately connected to the bank's VPN, trying to keep calm and shake off sleep brain fog as a sense of panic set in...

## 2.1 - Identify the compromised credentials

Now online and frantically loading the OpenShift Web Console for the `acme-prd` cluster along with additional tabs for observability tools Sarah keeps Morgan updated with her progress.

> Ok I'm logged in, do you want me to kill the pod? What do we do??

> Whatever you do, don't kill the pod yet!  Warns Morgan - we'll lose all the in memory evidence if we do that. We're monitoring the egress traffic and we'll isolate that shortly as our first actual counter move. For now I need you to collect some information. First up, how the hell are they executing this reverse shell? What credential are they using Sarah??  We need to know if this is bigger than just one pod or not.

> We'll need to track that down in the OpenShift Audit Logs Sarah stammers back, give me a few minutes to hunt through the logs.

Let's review the audit logs, specifically for pods within the `payment-gateway` namespace.  We can quickly query these via `oc`. In an incident situation, fast response is critical, having the skills and tooling to be able to quickly query audit logs is immensely valuable.

In the example below we'll query the audit logs held on one of our control nodes specifically for `exec` events occurring in the namespace of interest.

```bash
oc adm node-logs $(oc get nodes -l node-role.kubernetes.io/master --output name | head -n 1) \
  --path=kube-apiserver/audit.log \
  | jq 'select(.objectRef.namespace == "payment-gateway" 
        and .verb == "create" 
        and .objectRef.subresource == "exec")'
```

Take some time to review the audit log event and get familiar with the fields. **Note down the user that carried out the exec, we'll need this later.**

> Note: Querying node logs directly is only effective within the retention period for cluster logs, these files are rotated on disk. Forwarding logs to an external log store for long term retention is a vital security control.

## 2.5 Another pair of eyes

With everything happening so quickly Sarah opens a case with Red Hat support to confirm their initial findings around the use of a compromised credential and ensure she did not miss anything.

Let's run the following command to retrieve the audit logs directly from the cluster, into a directory that can be easily zipped and attached to a case at [https://access.redhat.com/support/cases/#/case/new](https://access.redhat.com/support/cases/#/case/new). For this workshop we won't actually open a case, however if this were a real event, quickly engaging expert assistance is highly recommended and we should practice the process of retrieving logs in a sharable, structured format.

```bash
oc adm must-gather -- /usr/bin/gather_audit_logs
```
