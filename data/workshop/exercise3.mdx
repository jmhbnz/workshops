## 3.1 Checkpointing the breached pod

Container checkpointing was introduced as an alpha feature in Kubernetes v1.25 and graduated to beta in Kubernetes v1.30 and is a [Generally Available feature in OpenShift 4.17 onwards](https://access.redhat.com/solutions/7008477). This functionality allows running containers to be transparently checkpointed to persistent storage and later restored to resume execution, or migrated across nodes and clusters.

The content of container checkpoints can be further analyzed with the `checkpointctl` tool. This allows to perform forensic analysis in case of security incidents (e.g., suspected compromise, data exfiltration) or application failures by inspecting the saved process memory, open files, sockets, and execution context captured in the checkpoint.

Initiate a checkpoint by copying and running the following script in your OpenShift Web Terminal:

```bash
NODE_NAME=$(oc get pods -l app=payment-gateway -o jsonpath='{.items[0].spec.nodeName}')
CURRENT_NS=$(oc project --short)

echo "==> Triggering fresh CRIU checkpoint on node: $NODE_NAME..."

API_RESPONSE=$(curl --silent --insecure \
  --header "Authorization: Bearer $(oc whoami --show-token)" \
  --request POST "${OPENSHIFT_API_URL}/api/v1/nodes/${NODE_NAME}/proxy/checkpoint/${CURRENT_NS}/payment-gateway/httpd")

TARBALL_PATH=$(echo "$API_RESPONSE" | jq -r '.items[0] // empty')

if [ -z "$TARBALL_PATH" ] || [ "$TARBALL_PATH" = "null" ]; then
  echo "[!] ERROR: Checkpoint failed. API response was:"
  echo "$API_RESPONSE"
  return 1 2>/dev/null || exit 1
fi

CHECKPOINT_FILE=${TARBALL_PATH##*/}

echo "==> Checkpoint created successfully: $CHECKPOINT_FILE"
echo "==> Streaming memory dump into Web Terminal..."

oc debug node/"$NODE_NAME" --to-namespace="$CURRENT_NS" -- \
  chroot /host /bin/sh -c "cat $TARBALL_PATH" > "/tmp/$CHECKPOINT_FILE"

echo "==> Success! Verifying saved forensic artifact in /tmp/:"
ls -lah "/tmp/$CHECKPOINT_FILE"
```

After a short wait you should see that a checkpoint has been created.
