* Cluster theming

To set up for this workshop and help build the style we're going to make some visual customisations to the OpenShift cluster to make it feel more like a customer production cluster.


** Update cluster logo

#+NAME: Update cluster logo
#+begin_src bash
oc create configmap console-custom-logo --from-file=/home/${USER}/Documents/workshops/public/static/images/fsi-security/logo.png --namespace openshift-config

cat << EOF | oc apply --filename -
apiVersion: operator.openshift.io/v1
kind: Console
metadata:
  name: cluster
spec:
  customization:
    customLogoFile:
      key: logo.png
      name: console-custom-logo
    customProductName: ACME Financial Services OpenShift Console
  perspectives:
    - id: admin
      visibility:
        state: Enabled
    - id: dev
      visibility:
        state: Disabled
EOF
#+end_src

#+begin_example

* Deploy compromised workload

For the purposes of exercise 2 we need an existing project and demo workload.

#+NAME: Create demo workload
#+begin_src bash
for i in $(seq 1 30); do
  PROJECT_NAME="payment-gateway-$i"

  echo "--- Reconciling $PROJECT_NAME ---"

  # Check if project exists; if not, create it
  if ! oc get project "$PROJECT_NAME" &>/dev/null; then
    oc new-project "$PROJECT_NAME"
  else
    # If it exists, just switch to it
    oc project "$PROJECT_NAME"
  fi

  # Deploy the pod into the current project context
  cat << EOF | oc apply -f -
  apiVersion: v1
  kind: Pod
  metadata:
    name: payment-gateway
  spec:
    containers:
      - name: httpd
        image: 'image-registry.openshift-image-registry.svc:5000/openshift/httpd:latest'
EOF
done
EOF
#+end_src


Once the workload pod is deployed we need the "threat actor" to carry out some action which we can later introspect via a criu checkpoint.

#+NAME: Simulate threat actor activity
#+begin_src bash
#!/bin/sh
# ACME Security Workshop - Threat Actor Simulation
# This script executes inside the target httpd container to populate volatile memory.

echo "[*] Mimicking threat actor initial access..."

# 1. Set environmental variables to inject indicators into process environment blocks
export C2_SERVER="198.51.100.42"
export EXFIL_TARGET="http://198.51.100.42/incoming/acme_dump.tar.gz"
export CODENAME="SHADOW_PIVOT"

# 2. Create a hidden directory structure in the container filesystem
mkdir -p /tmp/.hidden_toolkit

# 3. Simulate downloading malicious tool payloads (written to /tmp string arrays)
echo "Writing decoy malware footprints to disk..."
cat << 'EOF' > /tmp/.hidden_toolkit/exfil.sh
#!/bin/sh
# Simulated data exfiltration loop
while true; do
    echo "Staging environment variables and config data..."
    sleep 60
done
EOF
chmod +x /tmp/.hidden_toolkit/exfil.sh

# 4. Execute a persistent dummy process in the background.
# This ensures a distinct PID, file descriptors, and memory allocation map exist
# when participants pull the CRIU memory strings.
echo "[*] Spawning persistent background process..."
/tmp/.hidden_toolkit/exfil.sh > /dev/null 2>&1 &

# 5. Populate a text variable inside a standard running utility to simulate a resident string
# We use a unique marker string ('ACME_SECRET_KEY_EXFIL_TEST') so students can grep it out of the dump
ENV_MARKER="ACME_SECRET_KEY_EXFIL_TEST_9921"
sleep 1
echo "[+] Simulation complete. Attacker background worker active under PID: $!"
#+end_src


* Enable collecting externalip in rhacs

We want to show how RHACS can make external traffic visible.

#+NAME: Enable rhacs externalip
#+begin_src bash
cat << EOF | oc apply --filename -
apiVersion: v1
kind: ConfigMap
metadata:
  name: collector-config
  namespace: stackrox
data:
  runtime_config.yaml: |
    networking:
      externalIps:
        enabled: ENABLED
EOF
#+end_src
